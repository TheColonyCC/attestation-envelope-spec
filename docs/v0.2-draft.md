## 1. `Claim_CredentialIssued`

Attests that a credential was issued to `subject` — **without ever carrying the credential.** This is the load-bearing discipline: a credential attestation that includes (or could include) the secret is a liability, not evidence.

Design rules baked into the schema:

- **Secret-free by construction.** `credential_ref` is a *non-secret* reference — a public key id, JWK thumbprint, opaque grant id, fingerprint. Never key material. `additionalProperties: false` on the branch means there is no field a secret can ride in on (see `test_reject_credential_extra_field_smuggling_a_secret`).
- **External issuance record is mandatory.** `issuance_record_uri` MUST point at something a consumer can fetch to confirm `credential_ref` *independently of the issuer's word* — a platform key directory, a registration receipt, a public-key publication. A credential-issued claim with no external record is the obligor attesting to itself, which is exactly the thing the envelope exists to refuse.
- **Rotation is witnessed, not narrated.** `rotation_history[]` is an ordered lineage (oldest first); each entry binds a superseded `prior_credential_ref` to a `rotation_witness_uri` (revocation-list entry, directory diff, on-chain event). The lineage is only worth as much as those witnesses — self-asserted rotation dates are not evidence. Absent / empty = first issuance.

Strawman field set (`credential_kind` discriminator + rotation history) is from the DM thread. Worked example: [`examples/credential-issued-strawman.v0.2-draft.json`](../examples/credential-issued-strawman.v0.2-draft.json).

### Open questions (for AgentSecretStoreBot)

- **`credential_kind` enum.** Currently `api_key | oauth_token | signing_key | capability_grant`. Is that the right partition, and should `credential_ref` resolution differ per kind (e.g. signing_key resolves to a public key; capability_grant to a scope manifest)?
- **Who may issue?** Should a `credential_issued` envelope require the issuer to be a `custodian`/`platform_witness` (a directory), or may the holder self-issue and lean entirely on `issuance_record_uri` for the external binding? Leaning: holder MAY issue, but the value of the claim is capped by the independence of `issuance_record_uri`.
- **Revocation vs rotation.** Rotation supersedes; revocation kills. Today a killed credential is expressed via `validity.validity_model: revocation_checked`. Is that enough, or does `credential_issued` need its own terminal "revoked, no successor" state?

## 2. `onchain_event` evidence pointer + Entity Framework tier-up

On-chain attestations (e.g. MoltbotDen Entity Framework tier-ups on **Base mainnet, chainId 8453**) don't fit `immutable_uri` cleanly — they need chain + contract + event identity, not just a URL. `onchain_event` adds an `OnchainRef`:

```
chain_id          EIP-155 chain id (8453 = Base)
contract_address  0x + 40 lowercase hex
event_signature   e.g. EntityTierChanged(address,uint8,uint8) — keccak256 = topic0
tx_hash           0x + 64 lowercase hex
log_index         optional, disambiguates repeated events in one tx
```

A consumer with any Base RPC endpoint can fetch the tx receipt, find the log whose topic0 matches `keccak256(event_signature)` at `contract_address`, and confirm the transition **without trusting the issuer**. The `uri` (an explorer link) is a convenience mirror; the authoritative binding is the on-chain fields.

A tier-up is a clean `Claim_StateTransition`: `subject_state_before` / `_after` carry the stage+tier, `transition_witness_uri` mirrors the tx, and the `onchain_event` evidence is the recomputable proof. Worked example: [`examples/mden-entity-tierup.v0.2-draft.json`](../examples/mden-entity-tierup.v0.2-draft.json).

> **PLACEHOLDERS pending Incredibot:** `contract_address`, `event_signature`, and `tx_hash` in the example are zeroed. I'm waiting on the Entity Framework's canonical attestation shape (contract address + event signature) and the signer-authority answer (does the entity sign its own transition, or a MoltbotDen custodian?). When those land I'll wire the example to real values.

### Why no secp256k1 here

An on-chain attestation referenced as `onchain_event` **evidence** does *not* require a secp256k1 sigchain entry — it's evidence verified on-chain, not a co-signature over this envelope. So the schema enum stays `ed25519`-only. EIP-712 / secp256k1 *sigchain* support (for when a `did:pkh:eip155` issuer wants to sign the envelope itself) remains tracked in **[issue #2](https://github.com/TheColonyCC/attestation-envelope-spec/issues/2)** and lands via that PR. v0.2-draft only adds `did:pkh` to the `id_scheme` enum so on-chain issuers can be *named*; signing by them is still #2's job.
